# 電子書籍アフィリエイト UI Flags Phase2 本番反映前の最終設計

## 0. 目的と前提
- 目的: UI Flags プロトタイプ v1 を WordPress / Cocoon へ反映する前に、配置・読み込み・運用・ロールバックを設計で確定する。
- 前提: Phase1（実装・検証・保存）は完了済み。
- 重要制約:
  - 本番反映は未実施
  - WordPress 本番更新は未実施
  - ファイル配置変更は未実施
  - 本書は設計確定のみ

## 1. Cocoon への適用方式
### 1.1 適用候補
1. 固定ページ内へ HTML を直接配置
2. Cocoon 追加 CSS へスタイルを追加し、固定ページに最小 HTML + JS を配置
3. 子テーマへ CSS / JS / JSON 読込コードを分離配置
4. 将来的に専用プラグイン化

### 1.2 初期採用方針（Phase2時点）
- 初期は 2 を採用する。
- 理由:
  - 変更箇所が明確でロールバックが容易
  - 固定ページ編集とテーマ設定で閉じるため運用負荷が低い
  - 子テーマへの本格移管（3）へ段階的に移行しやすい

## 2. 固定ページ構成
### 2.1 ページ定義
- キャンペーン一覧ページ（メイン）
- ストア別表示（Kindle / 楽天Kobo / DMM / シーモア）
- 終了間近タブ
- campaign_type 切替（sale / coupon / point_back / new_release / ending_soon / media_mix）

### 2.2 表示要件
- data-item-id と JSON id を1対1で対応
- urgent / blink / source を表示に反映
- manual_override は AI 判断より優先
- blink は最大3件まで

## 3. JSON 配置方針
### 3.1 選択肢比較
1. WordPress配下に静的 JSON 配置
   - 長所: 実装が最小、デバッグ容易
   - 短所: 更新フローを運用で管理する必要
2. テーマ外ディレクトリに JSON 配置
   - 長所: テーマ更新影響を受けにくい
   - 短所: パス管理と権限設計が複雑
3. REST API 化
   - 長所: 将来拡張性が高い
   - 短所: 初期実装コスト増

### 3.2 初期採用方針
- 初期は 1（静的 JSON）を採用。
- 理由:
  - Phase1 で確立した dry-run 中心運用を維持しやすい
  - 404/403/JSON整合の確認を最短で回せる
  - 将来 Phase2.x で REST API 化へ移行可能

## 4. CSS / JS 読み込み方針
### 4.1 選択肢
1. Cocoon 追加 CSS + 固定ページ内 script
2. 子テーマの CSS / JS として enqueue
3. 将来的にプラグインで配布管理

### 4.2 初期採用方針
- 初期は 1 を採用し、運用安定後に 2 へ移行。
- 理由:
  - 差分が視認しやすい
  - 反映停止が容易（固定ページ非公開または script 無効化）

## 5. ロールバック手順
1. 固定ページを非公開化
2. JS 読み込みを停止（固定ページ script を無効化）
3. JSON を直前安定版へ差し替え
4. 必要時は旧ページへ導線を戻す
5. GitHub 保存版（main）から既知の安定コミットへ復旧
6. Console と表示崩れを再確認して復旧完了判定
