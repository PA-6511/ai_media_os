# AI_MEDIA_OS_GUI 危険操作モーダル仕様（確定版）

> 作成日: 2026-03-15  
> 前提画面: S3 接続/切断管理、S4 危険操作管理（画面構成確定版に準拠）

---

## 0. 仕様のゴール

- 認証と実行の境界を明確化し、誤操作を防止する
- 操作ごとに必要モーダル（通常確認 / 警告 / 認証 / 結果）を固定する
- 失敗時は fail-safe で必ず中断し、安全な状態に戻す
- 状態遷移表へ落とし込めるイベント・状態を定義する

---

## 1. 操作別の危険度分類

| 操作 | 実行画面 | 危険度 | 理由 |
|---|---|---|---|
| Connect | S3 `/connections` | 中 | 外部連携状態が変化するが可逆 |
| Disconnect | S3 `/connections` | 中 | 稼働系連携が停止するが可逆 |
| Freeze | S3 `/connections`（運用制御タブ） | 高 | 対象BlockAIの処理停止を伴い影響範囲が広い |
| Unfreeze | S3 `/connections`（運用制御タブ） | 中 | 停止解除で復旧操作、可逆 |
| Export | S4 `/danger-zone` | 重大 | データ持ち出しリスク、情報漏えいリスク |
| Delete | S4 `/danger-zone` | 最重大 | データ消失、取り消し不可 |

危険度レベル定義:
- 中: 2段階確認（通常確認 + 結果）
- 高: 3段階確認（通常確認 + 警告 + 結果）
- 重大/最重大: 5段階確認（通常確認 + 警告 + 認証(パスワード) + 認証(2FA) + 結果）

---

## 2. 操作ごとの確認ステップ

| 操作 | モーダルフロー | 実行可否の最終条件 |
|---|---|---|
| Connect | 通常確認 -> ハンドシェイク確認 -> 結果 | 送信側/受信側チェック両方ON + 対象有効 |
| Disconnect | 通常確認 -> ハンドシェイク確認 -> 結果 | 送信側/受信側チェック両方ON + 対象接続中 |
| Freeze | 通常確認 -> 警告 -> 結果 | 影響確認チェックON + 対象稼働中 |
| Unfreeze | 通常確認 -> 警告 -> 結果 | 復帰確認チェックON + 対象凍結中 |
| Export | 通常確認 -> 警告 -> 認証(パスワード) -> 認証(2FA) -> 結果 | すべての入力妥当 + 対象指定完了 |
| Delete | 通常確認 -> 警告 -> 認証(パスワード) -> 認証(2FA) -> 最終確認(対象名再入力) -> 結果 | すべての入力妥当 + 対象名完全一致 |

補足:
- Connect/Disconnect の「ハンドシェイク確認」は必須。バイパス不可。
- Export/Delete は認証完了前に API 実行不可（フロント・バック双方で拒否）。

---

## 3. 必要入力項目

| 操作 | 必須入力 |
|---|---|
| Connect | 操作対象、送信側確認チェック、受信側確認チェック |
| Disconnect | 操作対象、送信側確認チェック、受信側確認チェック |
| Freeze | 操作対象、凍結理由（必須テキスト）、影響確認チェック |
| Unfreeze | 操作対象、復帰確認チェック |
| Export | 操作対象、エクスポート範囲、再入力パスワード、6桁2FAコード、機密持ち出し確認チェック |
| Delete | 操作対象、削除理由（必須テキスト）、再入力パスワード、6桁2FAコード、対象名再入力、取消不可確認チェック |

入力ルール（共通）:
- パスワード: 空文字不可、最小8文字、送信前にマスク表示
- 2FAコード: 数字6桁固定、30秒タイムウィンドウ
- チェックボックス: 未チェック時は次へ進めない
- 対象名再入力: 完全一致（大文字小文字区別）

---

## 4. 実行前バリデーション

### 4.1 共通バリデーション

- 対象が未選択の場合はモーダル遷移不可
- 二重送信防止: 実行ボタン押下後は `pending` 解除まで再押下不可
- セッション有効期限切れ時は認証モーダルへ進ませずログインへ遷移
- 権限不足（role mismatch）は403として即中断（fail-safe）

### 4.2 操作別バリデーション

| 操作 | 実行前チェック | fail-safe中断条件 |
|---|---|---|
| Connect | 対象が切断状態、ハンドシェイクチェック2つON | 対象が既に接続中、ハンドシェイク不一致、タイムアウト |
| Disconnect | 対象が接続状態、ハンドシェイクチェック2つON | 対象が既に切断中、ハンドシェイク不一致、タイムアウト |
| Freeze | 対象が稼働中、凍結理由あり、影響確認ON | 対象が既に凍結、依存ジョブが保護状態 |
| Unfreeze | 対象が凍結中、復帰確認ON | 対象が未凍結、整合性チェックNG |
| Export | 対象あり、範囲妥当、パスワードOK、2FA OK | 認証失敗、対象不正、件数上限超過、監査ログ書込失敗 |
| Delete | 対象あり、理由あり、パスワードOK、2FA OK、対象名再入力一致 | 認証失敗、再入力不一致、参照整合性NG、監査ログ書込失敗 |

重要:
- Export/Delete は「監査ログ書込予約」の事前検証が失敗した時点で実行しない。
- Delete は参照整合性チェックに失敗した場合、強制削除オプションを表示しない。

## 5. 成功時/失敗時のUI表示

| 操作 | 成功時（結果モーダル） | 失敗時（結果モーダル） | 戻り先 |
|---|---|---|---|
| Connect | 「接続が完了しました」対象名、処理時間、監査ログID表示 | 「接続に失敗しました」原因、再試行可否、監査ログID表示 | S3（対象行をハイライト） |
| Disconnect | 「切断が完了しました」対象名、処理時間、監査ログID表示 | 「切断に失敗しました」原因、再試行可否、監査ログID表示 | S3（対象行をハイライト） |
| Freeze | 「凍結が完了しました」対象名、理由、監査ログID表示 | 「凍結に失敗しました」原因、依存情報、監査ログID表示 | S3（運用制御タブ） |
| Unfreeze | 「凍結解除が完了しました」対象名、監査ログID表示 | 「凍結解除に失敗しました」原因、監査ログID表示 | S3（運用制御タブ） |
| Export | 「Exportを開始しました/完了しました」対象、件数、監査ログID表示 | 「Exportを中断しました」原因（認証/権限/検証）、監査ログID表示 | S4（Exportセクション） |
| Delete | 「Deleteが完了しました」対象、削除件数、監査ログID表示 | 「Deleteを中断しました」原因（認証/再入力不一致/整合性）、監査ログID表示 | S4（Deleteセクション） |

失敗時の表示ルール:
- エラーコード + 人間向け説明を併記する
- 「どこまで通過したか」をステップ表示（例: 認証2FAまで成功、最終確認で失敗）
- 直前モーダルに戻さず、結果モーダルで完結して安全な画面に戻す

## 6. 監査ログに残す項目

### 6.1 全操作共通

- event_id（UUID）
- timestamp（UTC）
- operator_user_id
- operator_role
- source_screen（S3/S4）
- operation_type（connect/disconnect/export/delete/freeze/unfreeze）
- target_type（consensus_ai/block_ai/bulk）
- target_id / target_name
- request_id（フロント発行）
- result（success/failure/aborted）
- failure_reason_code（失敗時必須）
- fail_safe_triggered（true/false）
- ip_address
- user_agent

### 6.2 認証系追加項目（Export/Delete必須）

- password_reauth_result（success/failure）
- twofa_result（success/failure）
- twofa_method（totp/sms/other）
- auth_step_reached（password/twofa/final_confirm）

### 6.3 ハンドシェイク追加項目（Connect/Disconnect必須）

- handshake_required（true）
- handshake_sender_checked（true/false）
- handshake_receiver_checked（true/false）
- handshake_status（ok/timeout/mismatch）
- handshake_duration_ms

### 6.4 Delete/Exportの誤操作防止監査項目

- confirmation_phrase_required（true/false）
- confirmation_phrase_match（true/false）
- irreversible_ack_checked（true/false）
- preflight_validation_result（pass/fail）

## 7. モーダル文言のたたき台

### 7.1 Connect（通常確認 + ハンドシェイク）

- タイトル: 接続を開始しますか？
- 本文: 対象との接続状態を変更します。送信側・受信側の確認が必要です。
- チェック1: 送信側で接続命令を送信したことを確認しました。
- チェック2: 受信側で接続命令を受信したことを確認しました。
- 実行ボタン: ハンドシェイクを確定して接続

### 7.2 Disconnect（通常確認 + ハンドシェイク）

- タイトル: 切断を開始しますか？
- 本文: 対象との接続を停止します。影響範囲を確認してください。
- チェック1: 送信側で切断命令を送信したことを確認しました。
- チェック2: 受信側で切断命令を受信したことを確認しました。
- 実行ボタン: ハンドシェイクを確定して切断

### 7.3 Freeze（通常確認 + 警告）

- タイトル: 対象を凍結しますか？
- 本文: 凍結中は対象BlockAIの処理が停止します。
- 理由入力ラベル: 凍結理由（必須）
- チェック: 影響を確認し、凍結を実行することに同意します。
- 実行ボタン: 凍結を実行

### 7.4 Unfreeze（通常確認 + 警告）

- タイトル: 凍結を解除しますか？
- 本文: 凍結解除後、対象BlockAIは処理を再開します。
- チェック: 再開影響を確認しました。
- 実行ボタン: 凍結解除を実行

### 7.5 Export（通常確認 + 警告 + 認証）

- タイトル: Exportを実行しますか？
- 警告本文: この操作はデータ持ち出しを伴うため、再認証が必要です。
- 認証ラベル1: パスワードを再入力してください。
- 認証ラベル2: 2FAコード（6桁）を入力してください。
- チェック: 機密データ取扱ルールに従うことを確認しました。
- 実行ボタン: 認証してExportを実行

### 7.6 Delete（通常確認 + 警告 + 認証 + 最終確認）

- タイトル: Deleteを実行しますか？
- 警告本文: この操作は取り消せません。対象データは完全に削除されます。
- 認証ラベル1: パスワードを再入力してください。
- 認証ラベル2: 2FAコード（6桁）を入力してください。
- 最終確認ラベル: 対象名を正確に入力してください。
- チェック: この操作が取消不可であることを理解しました。
- 実行ボタン: 取消不可のDeleteを確定

文言ポリシー:
- 1文は40文字前後まで、重要語（取消不可/認証必須/影響）を先頭に置く
- 実行ボタン文言は動詞を明示（例: 実行、確定、開始）

---

## 8. 実装固定ルール（境界条件）

- 認証前に実行ボタンを有効化しない
- モーダルを途中で閉じた場合は必ず `aborted` として監査ログに残す
- 監査ログ書込に失敗した場合は業務操作を実行しない
- API側でも同じ検証を実施し、UI改ざんで回避できないようにする
- Delete/Export は同一セッションでの認証キャッシュを使い回さない（毎回再入力）

## 9. 状態遷移表作成に向けた最小状態定義

次工程で状態遷移表を作るため、以下の共通状態を採用する。

| 状態ID | 状態名 | 説明 |
|---|---|---|
| ST0 | Idle | 操作未開始 |
| ST1 | Confirming | 通常確認モーダル表示中 |
| ST2 | Warning | 警告モーダル表示中 |
| ST3 | AuthPassword | パスワード認証中 |
| ST4 | Auth2FA | 2FA認証中 |
| ST5 | Handshake | 相互確認ハンドシェイク中 |
| ST6 | FinalConfirm | 最終確認（対象名再入力など） |
| ST7 | Executing | API実行中 |
| ST8 | Success | 成功結果モーダル表示 |
| ST9 | Failed | 失敗結果モーダル表示 |
| STX | Aborted | ユーザー中断/Fail-safe中断 |

操作ごとの主経路:
- Connect: ST0 -> ST1 -> ST5 -> ST7 -> ST8
- Disconnect: ST0 -> ST1 -> ST5 -> ST7 -> ST8
- Freeze: ST0 -> ST1 -> ST2 -> ST7 -> ST8
- Unfreeze: ST0 -> ST1 -> ST2 -> ST7 -> ST8
- Export: ST0 -> ST1 -> ST2 -> ST3 -> ST4 -> ST7 -> ST8
- Delete: ST0 -> ST1 -> ST2 -> ST3 -> ST4 -> ST6 -> ST7 -> ST8

失敗/中断分岐（全操作共通）:
- 任意状態でバリデーション失敗 -> ST9
- 任意状態で権限/認証失敗 -> ST9
- 任意状態でユーザーキャンセル -> STX
- ST7で監査ログ失敗 -> ST9（業務操作は未実行）

## 10. Delete / Export 再確認（重点監査）

### Export の必須防止策

- 再認証（パスワード + 2FA）を毎回要求
- 対象範囲未指定で実行不可
- 機密取扱確認チェック未ONで実行不可
- 監査ログ予約失敗時は即中断

### Delete の必須防止策

- 再認証（パスワード + 2FA）を毎回要求
- 対象名再入力の完全一致必須
- 取消不可確認チェック未ONで実行不可
- 参照整合性NG時は実行不可
- 監査ログ予約失敗時は即中断

上記を満たさない実装は受け入れ不可とする。
