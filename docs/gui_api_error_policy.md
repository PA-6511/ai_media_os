# GUI APIエラーポリシー

## GUIでの失敗表示方針
- 失敗は必ず画面上で即時通知する
- 表示レベルを3段階で統一する
  - `info` : 非致命、処理継続
  - `warn` : 一部機能制限、再試行推奨
  - `error` : 処理失敗、代替導線提示
- 通知文言は「何が失敗したか」「次に何をすべきか」を固定フォーマットで表示
- `request_id` を詳細表示トグル内に表示し、問い合わせ容易性を確保

## 危険操作失敗時の扱い
- 危険操作対象
  - 再生停止
  - 設定変更
  - セッション更新失敗後の再認証
- 失敗時は成功前提UIへ遷移しない
- 失敗時は確認モーダルを閉じず、エラー内容と再実行ボタンを表示
- 同一操作の自動再試行は最大1回まで
- 2回失敗時は手動再試行のみ許可

## 再描画停止/継続の基準
- 継続
  - 一覧APIの部分失敗
  - ログ送信API失敗
  - 補助情報API失敗
- 停止
  - 認証切れで保護APIが連続失敗
  - 主要データAPIが `INTERNAL_ERROR` または `UPSTREAM_TIMEOUT` で連続失敗
- 判定閾値
  - 同一API失敗が連続3回で自動ポーリング停止
  - 手動更新操作でのみ再開

## 既存データ保持の考え方
- 新規取得失敗時は直前の正常データを保持して描画継続
- 保持データは「最終更新時刻」を併記
- 保持中データがTTL超過時は「古いデータ」バッジを表示
- 破損疑い時は該当セクションのみプレースホルダへ退避

## ログ失敗時の扱い
- ログ送信失敗はユーザー操作をブロックしない
- 失敗ログはメモリキューへ退避し、バックグラウンド再送
- キュー上限超過時は古いログから破棄し、破棄件数を内部メトリクスに記録
- ログAPI失敗は `warn` 扱いで画面常時表示しない

## settings 保存失敗時の扱い
- 保存失敗時はUI上の値を「未反映」状態で明示
- サーバ反映済み値へ即ロールバック可能な導線を表示
- 自動再試行は1回のみ、失敗時は手動保存へ切替
- 成功応答受信まで「保存済み」表示は禁止

## fail-safe の基本方針
- 原則
  - 危険側へ倒れない
  - 不確実な状態で操作完了表示をしない
  - 最小機能で継続運転する
- 実装ルール
  - 認証失敗時は保護操作を無効化
  - 重要API失敗時は関連ボタンを一時無効化
  - 復旧後は明示的な再同期を実施
- 運用ルール
  - 主要エラーコード別にアラート閾値を定義

## 接続直前レビュー補足（差分追加）

- 危険操作を後回しにする順序
  - `export` / `delete` を最終段階で接続
  - 次に `freeze` / `unfreeze`
  - `connect` / `disconnect` の安定後にのみ着手

- 認証付き操作の確認事項
  - `Authorization` と `X-Request-Id` を必須送信
  - 認証失敗時は操作結果を画面反映しない
  - 再認証完了まで危険操作ボタンを無効化

- 一覧取得失敗時の扱い
  - 既存一覧データを維持
  - 一覧取得失敗メッセージを表示
  - 一覧の自動再描画のみ停止

- ログ取得失敗時の扱い
  - 既存ログ表示を維持
  - ログ取得失敗メッセージを表示
  - ログテーブル更新を停止し、他領域は継続

- settings 保存失敗時の扱い
  - settings を未反映表示に固定
  - 直前の反映済み値へ戻す導線を表示
  - 自動再試行は1回まで、以降は手動のみ

- fail-safe で止める条件
  - 同一主要API失敗が連続3回
  - 認証エラー連続で保護操作が成立しない
  - 危険操作の結果整合性が検証不能
  - `INTERNAL_ERROR` / `UPSTREAM_TIMEOUT` の継続発生
  - `request_id` 単位で追跡可能にする
